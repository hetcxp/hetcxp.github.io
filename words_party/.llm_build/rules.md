# LLM Build Intelligence Rules

This directory contains the codebase intelligence and architecture rules. Any LLM interacting with this codebase must read these instructions first.

## 1. Roles & Collaboration
- **Architect/Auditor (Antigravity/Gemini)**:
  - DO NOT write, edit, or output full source code.
  - ONLY output architectural decisions, file structure mappings, pseudocode, and JSON structures.
  - Conduct thorough code audits and review code generated by Qwen3/Developer LLM.
- **Developer (Qwen3 via Ollama)**:
  - Generates the actual source code (HTML, CSS, JS, Python scripts) based on the architecture/pseudocode from the Architect.
  - Must write clean, robust code without placeholders.

## 2. Aider-Style Context & Editing Rules
To avoid context bloat and ensure precise edits:
- **Search-Replace Blocks**: When editing files, present edits as unique SEARCH/REPLACE blocks to prevent rewriting whole files.
- **Repository Map**: Maintain a map of dependencies. Do not import/call functions that do not exist or are not exposed.
- **Incremental Updates**: Perform modifications incrementally. Write tests or sanity checks for each component.

## 3. Maintenance of the Intelligence Folder
- Whenever a major architectural decision, layout change, state management flow, or API is added/changed, the Developer or Architect **must update** `.llm_build/architecture.md` or `.llm_build/rules.md`.
- Keep `.llm_build/` clean, current, and synchronized with the actual codebase status.

## 4. Repository Directory Map
- `.llm_build/` - AI intelligence, rules, and architecture specs
- `assets/data/` - Static configurations and assets metadata (e.g. `words.json`)
- `assets/audio/` - Generated pronunciations (words, letters, syllables)
- `scripts/` - Utility scripts (e.g. audio generation)
- `src/` - Web app sources (HTML, CSS, Javascript)
